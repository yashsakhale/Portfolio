DupZero

Project Motivation & Problem Statement

Emergency alert systems ingest thousands of notifications daily from news feeds, government agencies, social media, and IoT sensors. A critical operational challenge is that 40-60% of alerts describe the same underlying event with different wording, timestamps, and metadata. This duplication overwhelms human operators and delays critical response decisions. DupZero is a production-grade ML deduplication system that eliminates redundant alerts through a multi-stage pipeline combining rule-based filtering, semantic embeddings, clustering, and diversity-aware selection.

The core challenge: How do we accurately identify duplicate and near-duplicate alerts when they arrive from diverse sources with paraphrased content, different locations precision, and varying publication timestamps?

Technical Approach

1. Unified Service Architecture

  • Built a FastAPI application (main.py, version 0.2.0) exposing endpoints for classification (/classify), deduplication (/deduplicate), and web scraping (/scrape).
  • Implemented async lifecycle management: startup initializes embedding models, validates PostgreSQL schema, and warms up Playwright browser instances; shutdown gracefully releases resources.
  • Containerized with Docker for consistent deployment, with buildandpush.sh automating ECR pushes for AWS deployment.

2. Five-Stage Deduplication Pipeline

  • Stage 1 - Static Filter: run_full_static_filter() eliminates exact duplicates by messageId hash and filters stale alerts outside the configured cutoff window.
  • Stage 2 - Embedding Pipeline: run_embedding_pipeline() generates 1024-dimensional vectors using BAAI/bge-m3 (or 128-dim via Matryoshka truncation with all-MiniLM-L6-v2) and stores in pgvector-enabled PostgreSQL.
  • Stage 3 - DBSCAN Clustering: run_dbscan_dedup(eps=0.2, min_samples=2, metric='cosine') groups semantically similar alerts; refines clusters using exact GeoJSON coordinate matching for location-aware deduplication.
  • Stage 4 - MMR3 Diversity Selection: Enhanced Maximal Marginal Relevance with quality metrics, recency scoring, and 5Ws coverage analysis (check_5ws_coverage()) selects the most informative alert from each cluster (λ=0.5).
  • Stage 5 - Semantic Cross-Reference: run_semantic_dedup(threshold=0.2) compares incoming alerts against historical metadata table using pgvector's <=> cosine distance operator and PostGIS ST_Equals() for geo-filtering within configurable hours.

3. Embedding Infrastructure

  • Sentence-Transformers models loaded via get_model() with native dimension verification against NATIVE_DIMENSION constant.
  • Supports multiple models: bge-m3 (1024d), all-MiniLM-L6-v2 (384→128d via Matryoshka), all-mpnet-base-v2 (768d), e5-small/base (384/768d).
  • encode_texts() processes batches of 32 texts, applies L2 normalization after optional truncation, and returns float lists for database storage.
  • Embeddings stored in PostgreSQL stEmbedding column with pgvector extension for efficient similarity search.

4. Geo-Aware Deduplication

  • Alerts include areas JSONB field containing GeoJSON point coordinates for location-based filtering.
  • _extract_point_geom_sql() extracts coordinates via ST_SetSRID(ST_GeomFromGeoJSON(areas->0->'point'), 4326) for PostGIS spatial operations.
  • DBSCAN refinement step groups alerts by exact coordinate match within semantic clusters, preventing false deduplication of geographically distinct events.

5. MMR3 Quality Scoring

  • run_enhanced_mmr3_filter() scores alerts within clusters using composite metrics: summary quality, named entity extraction (spaCy NER when available), timestamp recency, and content completeness.
  • calculate_recency_score() applies exponential decay favoring recent alerts; compute_summary_metrics() validates informational coverage.
  • Results logged to file via log_results_to_file() for offline analysis and threshold tuning.

6. Web Scraping & Alert Enrichment

  • Two-stage extraction: stage_0_http_extractor.py for static HTML, stage_1_playwright_extractor.py for JavaScript-rendered content.
  • API endpoints: /scrape/extract (single URL), /scrape/batch (bulk), /scrape/alert (enrichment with classification).
  • /scrape/metrics exposes extraction success rates, latency percentiles, and failure categorization for operational monitoring.

Results

  • Production system processing alerts for Vigilant's emergency management platform with 99.7% uptime.
  • Five-stage pipeline reduced duplicate alerts by 58% compared to single-method approaches.
  • Geo-aware DBSCAN refinement eliminated false positives where semantically similar alerts described geographically distinct events.
  • MMR3 quality scoring ensured retained alerts had the most complete, recent, and informative content.

Limitations

  • Embedding model inference adds 50-200ms latency per batch; GPU acceleration not available in current deployment.
  • Matryoshka dimension truncation trades some semantic fidelity for storage efficiency—fine-tuning thresholds required per use case.
  • Playwright browser instances consume significant memory; concurrent scraping limited to prevent OOM conditions.
  • DBSCAN eps/min_samples tuning requires domain expertise—values optimized for emergency alerts may not generalize.
